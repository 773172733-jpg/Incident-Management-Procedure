# 事件树微信小程序
## 单人版 V1.0 产品与技术架构（预留团队协作版）

文档用途：作为 DeepSeek / Codex 的总开发蓝图。  
开发原则：**先完成单人版，底层从第一天预留团队字段与权限结构；不提前制作团队页面，不写无用的“假功能”。**

---

# 1. 产品定位

这不是普通便签，也不是只记录“今天做什么”的待办清单。

产品核心是：

> 用户创建一个“大事件”，把大事件拆成任意数量的小分支任务；每个分支可以单独设置时间和提醒；完成后勾选并变灰；所有分支完成后，由用户主动结束大事件；用户也可以提前结束大事件。

V1 服务个人用户。  
V2 扩展为团队任务指派：老板发布大事件给员工，员工自行拆分支，老板查看进度与完成日志。

暂定产品名：**事件树**。

---

# 2. V1 范围

## 2.1 必须实现

1. 微信用户初始化与个人数据隔离。
2. 创建、编辑、归档、删除、恢复大事件。
3. 大事件支持三种时间模式：
   - 不设置时间；
   - 设置起止期间；
   - 持续进行，无结束日期。
4. 在大事件中创建任意数量的分支任务。
5. 分支任务支持：
   - 名称；
   - 备注；
   - 分组；
   - 优先级；
   - 无时间、截止时间或起止期间；
   - 提醒；
   - 排序；
   - 完成、重新打开、删除。
6. 分支完成后变灰并显示完成时间。
7. 所有分支完成后提示用户是否结束大事件，但不自动结束。
8. 用户可以提前结束大事件。
9. 日历查看有时间的任务。
10. 动态页查看提醒、逾期事项和操作历史。
11. 已归档、回收站、基础设置。
12. 所有关键操作写入活动日志。

## 2.2 V1 不做

1. 团队创建与员工邀请。
2. 给其他微信用户指派任务。
3. 审批、退回、绩效统计。
4. 多人实时编辑。
5. AI 自动拆任务。
6. 聊天、评论、文件协同。
7. 付费与会员。
8. Web 管理后台。

这些能力只在字段、权限函数和日志结构中预留。

# 3. 核心概念

## 3.1 大事件 Project

大事件是一个可持续推进的目标，例如：

- 二级建造师考试；
- 新房装修；
- Python 学习；
- 张先生别墅瓷砖排版；
- 日本旅行计划。

大事件自身可以没有明确结束时间。

## 3.2 分支任务 Task

分支任务属于某个大事件，例如：

- 买教材；
- 完成法规第一章；
- 现场测量；
- 绘制客厅排版；
- 与客户确认。

V1 页面只展示“大事件 → 分支任务”两层。  
数据库保留 `parentTaskId` 和 `level`，以后可以支持任务下继续拆子任务，但 V1 不开放深层嵌套，避免界面和逻辑过度复杂。

## 3.3 分组 Group

每个大事件可以建立多个分组，例如：

- 教材学习；
- 刷题阶段；
- 模拟考试。

分组只是任务分类，不是新的任务层级。

## 3.4 动态 Activity

每次关键操作形成一条不可随意修改的记录，例如：

- 16:32 完成「卫生间排版」；
- 15:10 修改「客厅排版」截止时间；
- 11:06 添加分支「核对加工砖尺寸」；
- 09:20 提前结束大事件。

V1 给自己看；V2 直接成为老板查看员工进度的依据。

# 4. 时间设计

## 4.1 大事件时间模式

字段：`timeMode`

### none：不设置时间

- 不要求开始日期和结束日期。
- 首页显示“未设置时间”。
- 适合暂时不知道什么时候完成的事情。

### range：设置期间

- 必须有 `startAt` 和 `endAt`。
- `endAt` 不得早于 `startAt`。
- 首页显示“2026.07.01—2026.11.10”。
- 可计算剩余天数和是否超期。

### ongoing：持续进行

- 可设置 `startAt`，不设置 `endAt`。
- 首页显示“持续进行 · 已持续 N 天”。
- 适合学习、健身、长期投资记录等没有终点的事情。

## 4.2 分支任务时间模式

字段：`scheduleType`

### none

没有时间，不进入日历，不产生定时提醒。

### deadline

只有截止时间 `dueAt`。

### range

有 `startAt` 和 `endAt`，提醒默认围绕 `endAt`。

## 4.3 提醒规则

用户可选择：

- 准时提醒；
- 提前 10 分钟；
- 提前 30 分钟；
- 提前 1 小时；
- 提前 1 天；
- 自定义时间。

任务时间变更时，旧提醒必须取消并重新生成。  
任务完成、删除、随大事件结束时，所有未发送提醒必须取消。

# 5. 状态设计

## 5.1 大事件状态

字段：`status`

- `active`：进行中；
- `completed`：已结束；
- `archived`：已归档；
- `cancelled`：已取消，预留；
- `deleted` 不作为真实状态，删除使用 `deletedAt` 软删除字段。

## 5.2 分支任务状态

字段：`status`

- `todo`：未完成；
- `doing`：进行中，V1 可暂不提供单独按钮；
- `completed`：已完成；
- `closed_by_parent`：因大事件提前结束而关闭；
- `cancelled`：主动取消；
- `submitted`：员工已提交，等待老板验收，V2 预留；
- `approved`：老板验收通过，V2 预留；
- `rejected`：老板退回，V2 预留。

## 5.3 提前结束大事件

如果仍有未完成分支，弹窗明确提示：

> 当前还有 4 个未完成分支。结束大事件后，它们会显示为“随大事件结束”，不会算作已完成。

确认后：

1. 大事件状态改为 `completed`；
2. 未完成任务改为 `closed_by_parent`；
3. 每条被关闭任务保存 `statusBeforeParentClose`；
4. 写入“提前结束大事件”日志；
5. 实际进度仍显示“完成 5/9”，不得伪装成 100%。

## 5.4 重新打开大事件

重新打开后：

1. 大事件改回 `active`；
2. `closed_by_parent` 的任务恢复为 `statusBeforeParentClose`，没有旧状态时恢复 `todo`；
3. 原本真正完成的任务继续保持完成；
4. 写入重新打开日志。

# 6. 进度计算

进度不保存为唯一真值，应根据任务实时计算，必要时可保存缓存字段。

## 6.1 统计范围

排除：

- 已软删除任务；
- `cancelled` 任务；
- `closed_by_parent` 任务在“实际完成率”中不算完成。

完成状态包括：

- `completed`；
- `approved`。

公式：

`实际完成率 = 已完成有效任务数 / 全部有效任务数`

## 6.2 特殊情况

- 大事件没有分支：显示 0%，但允许手动结束。
- 所有有效分支完成：显示 100%，弹窗询问是否结束大事件。
- 提前结束：显示“已结束 · 实际完成 5/9”，不要显示假 100%。
- 持续项目可以长期保持 active，即使当前任务全部完成，也只提示，不强制结束。

- 全部；

整体参考已确认的设计稿风格，但不能直接复制其他产品。

# 9. 技术选型

## 9.1 前端

- 微信小程序原生；
- JavaScript；
- WXML；
- WXSS；
- 微信开发者工具；
- 不使用 uni-app；
- 不使用复杂前端框架；
- V1 不引入 Redux 等重型状态库。

选择原生 JavaScript 的原因：

- 用户已有微信小游戏开发经验；
- DeepSeek/Codex 修改成本较低；
- 上架链路简单；
- 方便以后使用微信原生能力。

## 9.2 后端

- 微信云开发 / CloudBase；
- 云数据库；
- 云函数；
- 云存储；
- 定时触发器；
- 小程序身份 OPENID。

## 9.3 安全原则

- 客户端不得提交或伪造 OPENID；
- 身份必须由云函数上下文取得；
- 所有新增、修改、删除都通过云函数；
- 读取也优先通过云函数统一鉴权；
- 不允许前端直接修改数据库关键字段；
- 所有时间使用服务端时间；
- 所有权限判断集中在后端。

# 11. 数据库设计

所有集合统一字段：

- `_id`
- `createdAt`
- `updatedAt`
- `deletedAt`
- `version`

`version` 用于以后处理多人同时修改和乐观锁。

---

## 11.1 users

```js
{
  _id: "数据库ID",
  openid: "微信OPENID",
  nickname: "",
  avatarUrl: "",
  status: "active",

  plan: "free",                 // V2 付费预留
  planExpiredAt: null,

  defaultReminderMinutes: 30,
  completedTaskSink: true,
  timezone: "Asia/Shanghai",

  createdAt: serverDate,
  updatedAt: serverDate,
  deletedAt: null,
  version: 1
}
```

索引：

- `openid` 唯一；
- `status + updatedAt`。

---

## 11.2 projects

# 14. 权限架构

即使 V1 只有自己，也必须从第一天集中写权限函数。

文件：`cloudfunctions/api/common/permission.js`

需要存在：

```js
canReadProject(userId, project)
canEditProject(userId, project)
canManageProject(userId, project)
canReadTask(userId, task, project)
canEditTask(userId, task, project)
canCompleteTask(userId, task, project)
canViewActivity(userId, activity)
```

## V1 规则

- 用户只能读取和修改自己 `ownerId` 的项目；
- `creatorId`、`assigneeId` 默认与当前用户相同；
- 前端传来的 ownerId、creatorId、assigneeId 一律忽略，由后端填写。

## V2 规则预留

- 创建者可看自己指派的项目；
- 负责人可更新允许更新的任务；
- 团队管理员可管理团队项目；
- 关注人只读；
- 是否允许负责人自行添加分支，由项目字段控制；
- 是否需要验收，由项目或任务字段控制。

不要把权限判断写在每个业务文件里反复复制。

# 15. 事务与一致性

以下操作必须尽量使用数据库事务，或通过可靠补偿方式保证一致：

## 15.1 完成任务

1. 校验权限；
2. 更新任务状态、完成时间和完成者；
3. 写活动日志；
4. 取消未发送提醒；
5. 更新项目进度缓存；
6. 检查是否所有有效任务完成；
7. 返回 `shouldPromptCompleteProject`。

## 15.2 提前结束大事件

1. 校验权限；
2. 更新项目；
3. 批量把未完成任务改为 `closed_by_parent`；
4. 取消这些任务的提醒；
5. 写项目日志；
6. 为每条任务写关闭日志，数量过大时可写一条批量日志并保存数量；
7. 更新缓存。

## 15.3 删除项目

软删除项目，同时软删除其任务、分组和未发送提醒。  
恢复项目时一起恢复。  
活动日志保留。

# 16. 排序设计

不要用数组下标作为永久顺序。

字段：`sortOrder`

初始间隔：

- 1000；
- 2000；
- 3000。

拖动到两个任务之间时取中间值。  
当间隔过小时，对当前分组重新编号。

完成任务自动下沉时，只改变显示排序或更新 sortOrder，不要破坏用户手动排序。

# 17. 缓存与页面刷新

V1 不引入复杂状态管理。

使用：

- `App.globalData` 保存用户基础信息；
- `event-bus.js` 发布轻量事件；
- 页面 `onShow` 做必要刷新；
- 创建或修改成功后发布：
  - `project:changed`
  - `task:changed`
  - `activity:changed`

首页、大事件详情、日历收到事件后刷新。

不能为了“看起来实时”而让所有页面频繁全量请求。

# 18. 提醒系统

## 18.1 两种提醒

### 小程序内提醒

最稳定。  
动态页显示到期、即将到期、逾期。

### 微信订阅消息

需要用户主动授权。  
只在获得有效订阅能力时发送。  
发送失败不得影响任务本身。

## 18.2 reminder-worker

定时执行：

1. 查询 `status = pending` 且 `triggerAt <= 当前时间`；
2. 原子修改为 `processing`，防止重复发送；
3. 判断任务是否仍有效、是否已完成；
4. 根据 channel 发送；
5. 成功改为 `sent`；
6. 失败改为 `failed` 或重试；
7. 写活动日志。

必须使用 `dedupeKey` 防重复。

## 18.3 V2 团队通知预留

员工完成任务时：

1. 永远写活动日志；
2. 创建通知事件；
3. 根据老板设置决定：
   - 每项立即提醒；
   - 每 30 分钟汇总；
   - 每日汇总；
   - 只提醒大事件完成；
   - 关闭外部提醒。

不要把“每勾一个分支必然发一条微信消息”写死。

# 22. 输入校验

后端必须再次校验，不能只依赖前端。

- 项目标题：1—40 字；
- 任务标题：1—60 字；
- 项目说明：最多 1000 字；
- 任务备注：最多 500 字；
- 分组名：1—20 字；
- 开始时间不得晚于结束时间；
- reminderOffsets 必须在允许范围；
- priority 必须在枚举中；
- status 不能由前端任意修改；
- sortOrder 必须是有限数字；
- 所有字符串去除首尾空格；
- 防止提交未允许字段。

# 23. 性能要求

1. 首页只请求卡片需要的字段。
2. 大事件详情任务分页加载，默认每次 30 条。
3. 活动日志分页加载。
4. 使用数据库索引，不做无索引大范围扫描。
5. 项目卡片使用进度缓存，关键操作后更新。
6. 云函数初始化尽量放模块顶部，避免重复创建数据库连接对象。
7. 避免每个任务单独请求一次。
8. 批量关闭任务使用批处理或事务。
9. 图片资源压缩，V1 尽量使用本地图标。
10. 骨架屏代替长时间空白。

# 26. 核心验收用例

1. 创建无时间大事件，首页正确显示“未设置时间”。
2. 创建有起止日期的大事件，日期校验正确。
3. 创建持续项目，正确显示持续天数。
4. 一个大事件创建 100 个任务，列表仍可正常加载。
5. 完成任务后变灰，记录完成时间。
6. 重新打开任务后恢复正常样式。
7. 所有任务完成，只提示结束，不自动结束。
8. 有未完成任务时提前结束，未完成任务显示“随事件结束”。
9. 提前结束后进度不显示假 100%。
10. 重新打开项目后，被父事件关闭的任务恢复。
11. 删除分组后任务进入“未分组”。
12. 修改任务时间后旧提醒取消。
13. 完成任务后提醒取消。
14. 删除项目后可在回收站恢复。
15. 用户 A 无法读取用户 B 的项目。
16. 前端伪造 ownerId 无效。
17. 重复点击完成不会生成多条冲突记录。
18. 提醒 worker 重复执行不会重复发送。
19. 活动日志中的旧标题快照不随任务改名而改变。
20. 无网络或云函数失败时给出明确提示，不丢失页面输入。

# 27. DeepSeek 编码纪律

1. 先读完整文档再生成代码。
2. 不擅自改变技术栈。
3. 不提前开发团队页面。
4. 所有数据写操作走云函数。
5. 不信任客户端传来的身份和状态。
6. 每个模块先给文件清单，再逐文件生成。
7. 每生成一个阶段，给出：
   - 新增文件；
   - 修改文件；
   - 数据库集合；
   - 索引；
   - 云函数部署步骤；
   - 测试步骤。
8. 不使用伪代码，不省略关键函数。
9. 不用“此处自行实现”。
10. 对已生成文件修改时，给完整文件或明确 diff，不能只描述。
11. 所有枚举集中管理。
12. 所有 UI 文案集中检查，保持中文。
13. 保持已确认的白底、橙色、圆角、轻卡片设计。
14. 任何团队预留只能体现在字段、权限和日志，不得让 V1 用户看到空团队入口。
15. 每阶段完成后先运行和修错，再进入下一阶段。

# 28. 最终产品原则
